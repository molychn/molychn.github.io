Vue源码学习10：数据响应式流程2

发表于 2019-07-09 | 更新于: 2019-07-09 | 分类于 计算机科学

上回书说到

上一篇有总结到Vue中关于数据的响应是如何做到的，我们都知道在定义一个对象时除了通过普通的赋值引用之外，还可以使用Object.defineProperty定义对象中的属性，其中还可以给每个属性增加getter/setter访问器。在数据响应过程中就用这种方式实现了数据获取与赋值时的更新。

除了对象有特定的方法之外，数组作为特殊的对象也有其监听的方法，套一层与数组具有相同方法的函数将原生的数组封装起来，起到一个中介监听的作用，从而实现监听。

在上一篇中使用了data属性举例，今天顺着data继续讲下去。简单回顾一下，在initData中末尾调用observe函数对data属性实现观察，通过观察observe的方法了解到它主要是为对象中的每个属性添加上ob属性，通过层层判断实例化一个Observer类，在类中遍历每个属性，包括数组的节点。

Observer类中调用了一个较为关键的函数defineReactive，它是实现Vue数据响应的关键。在它的定义中，首先获取原属性的getter与setter访问器，确保不会因其改变本身的方法，然后封装响应式getter/setter。在getter中有一段关键代码是用于收集依赖的，

通过判断Dep.target的存在收集依赖（关于target与depend稍后讲到），而后当属性被设置时触发setter访问器来执行这些被收集的依赖，

Observer-Watcher-Dep强强联合

已经知道Observer是用于对数据添加监听的，在监听观察的过程中还要顺手收集依赖与执行依赖，但是依赖是怎么来的呢，从字面上意思可以得出，Dep是主要负责收集依赖与执行的，先来看Dep类。

我将整个Dep类的内容都贴了上来，它的内容也不多，但却很关键，首先是构造函数，定义了实例的id属性，然后创建一个空数组用于存储依赖，上面我们说到依赖的收集使用的是depend方法，而Dep类的方法也只有4个，addSub方法是将每个Watcher类push到实例中的subs数组中，而有addSub就有removeSub用于清除。

depend中是这样收集依赖的，

判断Dep.target是否存在将自身实例添加到Dep.target中，Dep.target本身是一个Watcher类，其addDep千丝万缕最后还是回到Dep类中。

notify方法就是收集到的依赖的执行了，

也就是通过遍历执行数组中每个元素的update方法，而每个元素说到底它还是个Watcher类。

在dep的文件中最后有关于Dep.target的定义与赋值，然后还有pushTarget/popTarget用于target的进出。

大致讲完Dep类后需要了解一下关系密切的Watcher类，Watcher类传递5个参数，

  1. 实例Vue实例vm
  2. 表达式或函数
  3. 回调函数
  4. option参数对象
  5. 是否是渲染的Watcher类

首先将实例vm赋值给当前实例，后判断是有是渲染的Watcher类决定是否将当前实例赋值给watcher，之后将当前实例push到watchers数组中。再之后根据传递进来的option参数定义实例中的deep/user/lazy/sync/before属性，接下来一个比较关键的是定义了一套dep相关的属性，

这几个属性就是用于watcher过程中的执行以及避免重复执行的关键。在之后就是将expOrFn赋值给当前的getter，最后在末尾定义value属性，其值是根据lazy的属性来判断的，即如果非惰性的情况下，调用实例的get方法获取value值。

get与其他几个方法是Watcher类中定义的，分别有，

  1. addDep，Add a dependency to this directive
  2. cleanupDeps，Clean up for dependency collection
  3. update，Will be called when a dependency changes
  4. run，Will be called by the scheduler
  5. evaluate，This only gets called for lazy watchers
  6. depend，Depend on all deps collected by this watcher
  7. teardown，Remove self from all dependencies’ subscriber list
  8. get，Evaluate the getter, and re-collect dependencies

get是Watcher实例的过程会被执行的方法，

get首先是调用了全局的pushTarget方法，这算是整个源码中对pushTarget(this)传递了this的地方，调用了pushTarget(this)，此时Dep.target的值为当前这个实例，而后value的值通过实例化过程中定义的getter执行获得，

当然这其中使用了try-catch来捕获错误，如果错误的话，根据user属性值报相应的错误提示，最后在finally块中根据deep值决定是否执行traverse函数，这里主要是用于深度监听的作用，最后调用popTarget将当前实例从target栈中推出，然后执行cleanupDeps方法。

cleanupDeps注释中就说明了它是用于清除收集的依赖的，

使用while循环将实例中的dep数组根据newDepIds是否已经存在执行removeSub方法移除依赖，而后将new属性下的值赋值给原先的，然后清空new属性相关值。

其余的方法又是什么时候执行的呢，讲完三类强强联合，我们需要回到流水线上来研究到底是如何执行依赖的收集与执行的。回到的Vue实例的init初始化操作，在整个_init方法中末尾调用了vm.$mount方法，这个方法就是实现渲染render与数据观察的关键。

排除Vue实例中其他属性的监听，暂且不提，以后的补足中慢慢的查缺补漏，我们继续以data为主。首先调用vm.$mount我们需要了解发生了什么。$mount的定义在两个文件中有涉及，/platforms/web/runtime/index.js以及/platforms/entry-runtime-with-compiler.js，也就是其中一个集成了compiler，但两者的共同点都是用于调用mountComponent方法。

mountComponent就是进行render渲染，在beforeMount与mounted两个callhook钩子的中间有一个关于updateComponent的定义，它的目的就是执行vm._render方法，而后就是实例化Watcher类了，

向Watcher传递实例vm，updateComponent则会作为类中的getter在get方法中执行，noop代表为空，然后传递了option参数，在参数中包含了一个before方法，其中里面有个beforeUpdate的callhook生命周期钩子，也就表明，在beforeMount后，会先进入到beforeUpdate周期中，最后传递一个true，也就表明当前实例的Watcher类是用于渲染的，执行到最后再执行mounted生命周期钩子。

响应式流程

也就是在这个Watcher类中，通过执行updateComponent会执行vm实例中_render方法，而进行渲染就需要对模版上的各个数据节点取值，取值则又触发getter方法，从而执行到我们在Observer类相关内容中的getter与setter属性。

在Watcher类中的get方法因为执行了

所以我们在Observer相关内容中getter就可以收集相应的依赖，

在收集依赖的过程中则会执行dep.depend方法，

Dep类的depend就是将当前实例添加到Dep.target中，

Dep.target就是执行updateComponent时，get方法pushTarget进来的watch实例，所以addDep，

判断当前实例中newDepIds是否存在该dep，没有的话就向newDepIds，newDeps添加进去，如果depIds也不存在该dep的话，则执行dep的addSub方法，也就是将当前watcher实例添加到dep的subs数组中，

执行到这一步基本上已经将属性的依赖收集个七七八八了，然后我们接下来需要做的就是怎么去执行这些依赖，还是要回到Observer类相关内容中，定义的setter访问器监听到属性的值有变化时，开始执行依赖，

dep的notify拿到this.subs数组中的第一元素，然后有个判断，即在非生产的情况下如果是非异步的，则对subs进行排序，最后开始遍历subs，执行其中每一个元素的update方法，因为存在subs中的就是watch实例，所以update，

执行update的过程需要根据lazy与sync的值来判断执行的方法，如果lazy为true，则先将dirty赋值为true，然后根据sync的值决定执行run还是queueWatcher方法，默认来说sync为false，所以会执行queueWatcher方法，其中涉及到nextTick的方法，这里先不了解，其中在这个过程中会执行watch.run方法，也会执行updated的生命周期钩子，

run方法就是根据active又将get执行了一遍，在get中最后是会执行cleanupDeps方法的，也就是将执行过的依赖清除掉，最后因为value值的改变，会执行cb回调函数。
